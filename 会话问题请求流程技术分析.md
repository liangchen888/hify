# Hify 会话问题请求流程技术分析

## 一、核心流程图

```
用户提交问题 (UI)
    ↓
POST /api/v1/chat/sessions/{sessionId}/messages/stream (或 /messages)
    ↓
ChatController.streamMessage() / sendMessage()
    ↓
ChatService.streamChat() / syncChat()
    ↓
┌─────────────────────────────────────────────────────────────┐
│  工程内 API 调用链路                                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1️⃣ 会话/Agent配置加载 (工程)                               │
│     ↓                                                        │
│     chatSessionMapper.selectById(sessionId)                 │
│     agentMapper.selectById(agent_id)                        │
│     [读Agent配置: model, systemPrompt, tools, knowledge]    │
│                                                              │
│  2️⃣ 多轮对话上下文加载 (工程 + Redis)                       │
│     ↓                                                        │
│     Redis (key: session:{sessionId})                        │
│     → 若缓存未命中，从 MySQL chat_message 表查询              │
│     → 加载最近 maxContextTurns*2 条消息                     │
│                                                              │
│  3️⃣ RAG 知识库检索 (工程 + pgvector)                        │
│     ↓                                                        │
│     IF agent.knowledgeBaseId != NULL:                       │
│       knowledgeService.searchChunks(kbId, userContent, 3)   │
│       → 向量化用户输入                                       │
│       → pgvector 语义相似度检索 Top 3 chunks                │
│       → 构建引用片段到 SystemPrompt                          │
│                                                              │
│  4️⃣ MCP 工具Schema加载 (工程)                              │
│     ↓                                                        │
│     agentToolMapper.selectMcpServerIdsByAgentId(agentId)    │
│     FOR EACH mcpServerId:                                   │
│       mcpService.listToolsDetail(serverId)                 │
│       → 调用 MCP Server 获取工具列表、description、schema   │
│       → 构建 OpenAI tools format JSON                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│  大模型 LLM 判断能力 (外部 API 调用)                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  🤖 第一轮 LLM 调用: 问题理解 + 工具判断                    │
│     ↓                                                        │
│     ProviderAdapter.streamChat(provider, chatRequest)       │
│       → 发送到 OpenAI / Claude / Gemini / Ollama ...        │
│       → 消息体包含:                                         │
│          - SystemPrompt (Agent配置 + RAG引用)              │
│          - 多轮对话历史 (从工程加载)                        │
│          - 用户当前问题                                     │
│          - MCP 工具Schema列表 (来自工程)                    │
│       → LLM 判断: 是否需要调用工具？                        │
│       → 如果 finishReason = "tool_calls":                   │
│          返回要调用的工具列表及参数                          │
│       → 如果 finishReason = "stop":                         │
│          返回最终答案，流程结束 ✓                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
    ↓
    ├─ 如果无需工具调用 (finishReason == "stop")
    │   ↓
    │   ✓ 保存助手回复 → MySQL
    │   ✓ 更新Redis上下文缓存
    │   ✓ 返回流式/同步结果给客户端
    │   ✓ 流程结束
    │
    └─ 如果需要工具调用 (finishReason == "tool_calls")
        ↓
┌─────────────────────────────────────────────────────────────┐
│  工具执行循环 (工程 + MCP Server)                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  FOR EACH toolCall in llmResponse.toolCalls:                │
│                                                              │
│    🔧 工具执行 (工程调用 MCP Server)                        │
│       ↓                                                      │
│       mcpService.callTool(serverId, toolName, args)        │
│         → DirectMcpClient.call() 发送 HTTP 请求           │
│         → 到 MCP Server endpoint (e.g. /mcp)             │
│         → 执行真实业务逻辑 (订单查询、退款等)             │
│         → 返回工具执行结果 (JSON)                         │
│                                                              │
│       [Fallback: 若 MCP Server 不可达, 返回 mock 数据]   │
│                                                              │
│    结果添加到消息列表:                                       │
│       ↓                                                      │
│       messages.add({                                        │
│         role: "tool",                                       │
│         tool_call_id: toolCall.id,                          │
│         content: toolResult  // 工具执行结果              │
│       })                                                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────────────────────────┐
│  第二轮 LLM 调用: 工具结果理解 + 最终响应                   │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  🤖 再次调用 LLM                                            │
│     ↓                                                        │
│     messages = [                                            │
│       {role: "system", content: systemPrompt},             │
│       {role: "user", content: userQuestion},               │
│       {role: "assistant", content: "", toolCalls: [...]},  │
│       {role: "tool", tool_call_id: "...", content: result},│
│       {role: "tool", tool_call_id: "...", content: result},│
│       ...                                                   │
│     ]                                                       │
│       → LLM 理解工具结果，生成最终答案                      │
│       → finishReason = "stop" (通常不再需要工具)           │
│       → 返回最终回复内容                                    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
    ↓
    ✓ 保存最终回复 → MySQL
    ✓ 更新Redis上下文缓存
    ✓ 返回完整结果给客户端
    ✓ 流程结束
```

## 二、技术调用链路详细分析

### A. 工程内 API 调用部分

#### 1. 会话/Agent配置加载

**调用链**:
```java
ChatServiceImpl.streamChat(sessionId, request)
  ↓
ChatServiceImpl.doStreamChat()
  ↓
chatSessionMapper.selectById(sessionId)           // MySQL 查询
agentMapper.selectById(session.agentId)           // MySQL 查询
  ↓
modelConfigMapper.selectById(agent.modelConfigId) // MySQL 查询
providerMapper.selectById(modelConfig.providerId) // MySQL 查询
```

**关键数据**:
- Agent.systemPrompt: 用于构建系统提示词
- Agent.modelConfigId: 指定使用的模型
- Agent.knowledgeBaseId: 是否启用 RAG
- Agent.workflowId: 是否启用工作流（若有则不走 LLM 路径）
- Agent.maxContextTurns: 多轮对话历史轮数

---

#### 2. 多轮对话上下文加载

**调用链**:
```java
ChatServiceImpl.loadContext(sessionId, maxMsgs)
  ↓
IF Redis(key: session:{sessionId}) 命中:
  返回 Redis 缓存的消息对象
ELSE:
  chatMessageMapper.selectRecent(sessionId, maxMsgs)  // MySQL 查询
  构造消息格式返回
```

**工作原理**:
- Redis TTL: 2 小时
- 采用 Cache-Aside 模式
- 数据格式: List<Map<String, String>>，包含 role/content
- 用途: 提高性能，减少数据库查询

---

#### 3. RAG 知识库检索

**调用链**:
```java
IF agent.knowledgeBaseId != null:
  knowledgeService.searchChunks(kbId, userContent, 3)
    ↓
    documentChunkMapper.searchByVector(kbId, vector, 3)  // pgvector 查询
      ↓
      使用 pgvector 的向量相似度计算
      SELECT * FROM document_chunk 
      WHERE kb_id = ? 
      ORDER BY embedding <-> embedding(?) LIMIT 3
    ↓
    返回 Top 3 相似 chunks
```

**技术细节**:
- 使用 pgvector 插件进行向量化存储和检索
- 调用 embedding 模型将用户输入向量化（通常是 OpenAI 的 embedding 模型）
- 返回的 chunks 会被组装成 RAG 引用片段，插入到 SystemPrompt

**示例输出**:
```
相关知识库片段:
1. "退款政策：7天无理由退货..."
2. "物流查询API：/api/logistics/track..."
...
```

---

#### 4. MCP 工具Schema加载

**调用链**:
```java
ChatServiceImpl.buildToolSchemas(agentId)
  ↓
agentToolMapper.selectMcpServerIdsByAgentId(agentId)  // MySQL 查询
  ↓
FOR EACH mcpServerId:
  mcpServerMapper.selectById(serverId)               // MySQL 查询
  mcpService.listToolsDetail(serverId)
    ↓
    DirectMcpClient.call("tools/list")
      ↓
      HTTP POST 到 MCP Server: http://{host}:{port}/mcp
      请求体: { "jsonrpc": "2.0", "method": "tools/list", ... }
    ↓
    返回工具列表
    [
      {
        name: "query_order",
        description: "查询订单详情",
        inputSchema: { type: "object", properties: {...} }
      },
      ...
    ]
  ↓
  构建 OpenAI Function Calling 格式:
  {
    type: "function",
    function: {
      name: "query_order",
      description: "查询订单详情",
      parameters: {type: "object", properties: {...}}
    }
  }
```

**工作原理**:
- 通过 MCP Server 的 HTTP 接口获取可用工具列表
- 将工具元数据转换为 LLM 能理解的格式（OpenAI tools format）
- 若 MCP Server 无法连接，使用 fallback 机制返回预定义的工具列表

---

### B. 大模型 LLM 判断能力部分

#### 第一轮 LLM 调用

**调用链**:
```java
ProviderAdapter.streamChat(provider, chatRequest)
  ↓
根据 provider.type 选择具体实现:
  - OpenAIAdapter → call OpenAI API
  - ClaudeAdapter → call Anthropic API
  - GeminiAdapter → call Google API
  - OllamaAdapter → call Ollama local API
  ↓
发送 HTTP 请求到 LLM API
```

**请求体构成**:
```json
{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "system",
      "content": "你是一个客服助手。\n\n相关知识库:\n[RAG检索的chunks]\n\n可用工具:\n- query_order: 查询订单\n- submit_refund: 提交退款..."
    },
    {
      "role": "user",
      "content": "用户输入的问题"
    },
    {
      "role": "user",
      "content": "上轮对话消息..."
    },
    ...
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "query_order",
        "description": "查询订单详情",
        "parameters": {...}
      }
    },
    ...
  ],
  "temperature": 0.7,
  "max_tokens": 2000
}
```

**LLM 判断过程** (大模型的能力):
1. **理解意图**: 根据 system prompt + 历史对话 + 用户问题，理解用户需求
2. **决策**: 判断是否需要调用工具
   - 若需要：返回 `finish_reason: "tool_calls"` + toolCalls 列表
   - 若不需要：返回 `finish_reason: "stop"` + 直接的文本答案
3. **参数生成**: 若选择调用工具，生成工具名和参数

**示例 LLM 响应**:
```json
{
  "finish_reason": "tool_calls",
  "tool_calls": [
    {
      "id": "call_123",
      "type": "function",
      "function": {
        "name": "query_order",
        "arguments": "{\"order_id\": \"20240501001\"}"
      }
    }
  ],
  "content": null
}
```

---

#### 工具执行（工程调用外部系统）

**调用链**:
```java
FOR EACH toolCall:
  ChatServiceImpl.executeToolCall(toolCall, agentId)
    ↓
    从 Agent 绑定的 MCP Server 列表中查找
    ↓
    mcpService.callTool(serverId, toolName, args)
      ↓
      DirectMcpClient.call(toolName, args)
        ↓
        HTTP POST 到 MCP Server
        URL: http://{mcp-server-host}:{port}/mcp
        请求体: {
          "jsonrpc": "2.0",
          "method": "tools/call",
          "params": {
            "name": "query_order",
            "arguments": "{\"order_id\": \"20240501001\"}"
          }
        }
      ↓
      MCP Server 执行业务逻辑 (调用真实的订单系统API)
      ↓
      返回结果: {"order_status": "已发货", "tracking": "..."}
    ↓
    若调用失败 (MCP Server 不可达):
      使用 buildMockToolResult() 返回预定义的模拟数据
```

**特点**:
- 工程不直接调用业务系统，而是通过 MCP Server 作为中介
- MCP Server 本身可能调用真实的后端服务（订单系统、物流系统等）
- 失败时自动降级到 mock 数据，保证对话不中断

---

#### 第二轮 LLM 调用

**调用链**:
```java
消息列表添加工具结果:
messages.add({
  role: "tool",
  tool_call_id: call_123,
  content: "{\"order_status\": \"已发货\", \"tracking\": \"...\"}"
})
// 可能有多个工具结果

再次调用 LLM:
ProviderAdapter.streamChat(provider, chatRequestWithToolResults)
  ↓
LLM 理解工具结果，生成最终答案
```

**请求体构成**:
```json
{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "system",
      "content": "你是客服助手..."
    },
    {
      "role": "user",
      "content": "我想查询订单20240501001的状态"
    },
    {
      "role": "assistant",
      "content": null,
      "tool_calls": [
        {
          "id": "call_123",
          "function": {
            "name": "query_order",
            "arguments": "{\"order_id\": \"20240501001\"}"
          }
        }
      ]
    },
    {
      "role": "tool",
      "tool_call_id": "call_123",
      "content": "{\"order_status\": \"已发货\", \"tracking\": \"SF123456\"}"
    }
  ],
  "tools": [...],  // 工具定义仍然存在，允许 LLM 继续调用
  "temperature": 0.7,
  "max_tokens": 2000
}
```

**LLM 生成最终答案** (大模型的能力):
- 根据工具返回的数据，生成人类易读的回答
- 通常返回 `finish_reason: "stop"`
- 返回最终的文本答案

**示例最终回复**:
```
您的订单（订单号：20240501001）已于昨天发货，物流单号是 SF123456。
您可以通过顺丰官网查询详细物流信息。
```

## 三、调用链路概览表

| 阶段 | 组件 | 调用方式 | 作用 |
|------|------|--------|------|
| **1. 加载配置** | Agent/ModelConfig/Provider | MySQL 查询 | 确定模型、工具、知识库配置 |
| **2. 加载上下文** | Redis + MySQL | 缓存查询 + 数据库 | 获取多轮对话历史 |
| **3. RAG 检索** | pgvector | 向量相似度检索 | 获取相关知识片段 |
| **4. 工具Schema** | MCP Server HTTP API | 跨进程调用 | 获取可用工具列表 |
| **5. 第一轮LLM** | OpenAI/Claude/Gemini | HTTP 外部API | **LLM 判断是否需要工具** |
| **6. 工具执行** | MCP Server HTTP API | 跨进程调用 | 执行具体业务逻辑 |
| **7. 第二轮LLM** | OpenAI/Claude/Gemini | HTTP 外部API | **LLM 理解工具结果，生成答案** |
| **8. 持久化** | MySQL + Redis | 数据库写入 + 缓存 | 保存对话记录和上下文 |

## 四、哪些是工程 API，哪些是 LLM 能力

### ✅ 工程内 API 调用

1. **Agent 配置加载** - 工程 Mapper 直接操作 MySQL
2. **上下文管理** - Redis Cache-Aside 模式
3. **RAG 向量检索** - pgvector 插件查询
4. **工具列表获取** - 通过 HTTP 调用 MCP Server（跨进程）
5. **工具执行** - 通过 HTTP 调用 MCP Server 执行业务逻辑
6. **对话持久化** - MySQL INSERT/UPDATE
7. **消息流编码** - SSE 事件推送

### 🤖 LLM 大模型能力

1. **意图理解** - LLM 根据 system prompt 和历史理解用户需求
2. **工具决策** - LLM 判断是否需要调用工具、选择哪个工具
3. **参数生成** - LLM 为选定的工具生成合理的参数
4. **结果理解** - LLM 理解工具返回的数据
5. **答案生成** - LLM 用自然语言生成最终回复

## 五、请求流程时序图

```
用户 →→ UI →→ 工程 →→ LLM →→ 工程 →→ MCP Server →→ 业务系统

t0  提交问题 POST /chat/sessions/{id}/messages/stream
    |
t1  加载 Agent、Model 配置 (MySQL)
    |
t2  加载上下文 (Redis/MySQL)
    |
t3  RAG 检索 (pgvector)
    |
t4  获取工具 Schema (MCP Server HTTP)
    |
t5  [第一轮] 发送给 LLM
    |                    🤖 LLM 处理
t6  返回: tool_calls / final_answer
    |
    ├─ 若无需工具: 保存回复 → 返回给用户 ✓ END
    |
    └─ 若需要工具:
       |
       t7 FOR EACH toolCall:
          |
          t8 调用 MCP Server (HTTP) → 业务系统
             |
          t9 返回工具结果
          |
       t10 组装工具结果到消息
           |
       t11 [第二轮] 发送给 LLM (含工具结果)
           |                    🤖 LLM 理解结果，生成答案
       t12 返回最终答案
           |
       t13 保存回复 (MySQL)
           |
       t14 更新上下文缓存 (Redis)
           |
       t15 返回给用户 ✓ END
```

## 六、架构总结

### 工程职责
- **配置管理**: Agent、Model、MCP Server 配置维护
- **编排**: 控制请求流程、工具调用顺序、上下文管理
- **适配**: 多模型提供商适配（OpenAI、Claude、Gemini、Ollama）
- **存储**: MySQL 数据持久化、Redis 缓存、pgvector 向量库
- **集成**: 通过 MCP Server 桥接业务系统

### LLM 职责
- **理解**: 理解用户意图和历史上下文
- **决策**: 判断是否调用工具、选择哪个工具
- **生成**: 生成工具参数和最终答案

### 关键设计
1. **工具中介化**: 工程不直接调用业务系统，通过 MCP Server 中介，降低耦合
2. **失败降级**: MCP Server 不可达时使用 mock 数据，保证对话流畅
3. **异步流式**: SSE 长连接实时推送，提升用户体验
4. **缓存优化**: Redis 缓存上下文和配置，减少数据库访问
5. **可观测性**: 详细的日志和指标埋点，便于问题追踪
